# Travel Guide

## 目标

生成一份可执行的个性化旅游攻略，并写入 Notion Database。攻略必须覆盖 `dimensions.md` 的全部维度，不能泛泛而谈。

## 协作 Skill

按顺序使用：

1. `travel-intake`：确认偏好、硬约束、出发与返程边界。
2. `travel-research`：采集小红书、高德地图、飞猪数据，并保留来源链接。
3. `travel-publisher`：组织攻略、写入 Notion、执行发布前验收。

## 必须遵守

- 未确认出发和返程时间点，不得编排每日行程。
- 先锁定大交通，再安排两端接驳、酒店、每日游玩。
- 以核心景区/活动为中心分组，再向外筛选酒店、餐厅和通勤。
- 机场/车站、酒店、景区、餐厅之间的关键移动必须用高德地图 MCP 给出方式、距离、预计时间。
- 酒店、票务、交通产品、核心小红书笔记必须附跳转链接。
- Notion Database 使用画廊视图；入选景区、餐馆、酒店尽量配图并保留图片来源。
- 天气预警、高温暴雨台风等风险必须给出室内备选方案。
- 飞猪数据只能直接调用 `flyai-cli`，不得依赖外部 `flyai` skill。

## 工作流

1. 读取 `dimensions.md`，建立必填维度检查表。
2. 用 `travel-intake` 产出结构化需求摘要。
3. 用 `travel-research` 产出带来源的候选景点、活动、酒店、餐厅、交通和风险信息。
4. 按地理聚类与时间边界生成每日方案。
5. 用 `travel-publisher` 写入 Notion 并输出页面链接。
6. 最后汇报：Notion URL、核心路线、预算、关键预订链接、未确认事项。

## 质量门禁

发布前逐项确认：

- `dimensions.md` 的每项均有结论、行动或来源。
- 每日行程含上午、午餐、下午、晚餐、夜间/休息、住宿。
- 每段关键通勤含出发地、目的地、方式、距离、时间。
- Notion 页面有封面图，景区/餐馆/酒店卡片有图片或明确补图标记。
- 每个推荐都说明为什么匹配用户偏好。
- 所有价格标注“参考价格，以实时为准”。
